# press-fleet 架构

> 中文为主（目标读者是中文创作者），术语保留英文。英文概览见 README.md。

## 一句话

press-fleet 是一支跑自媒体的 AI 编辑舰队：**信号进 → 六角色流水线 → 确定性门禁 → 发布 → 复盘 → SOP 改版**，全程明牌，SOP 本身开源、带版本号、接受读者反馈迭代。

## 与 research-fleet 的关系

同一套哲学的姊妹项目：research-fleet 给科研项目配一支不偷工减料的 AI 团队（每个数字可审计），press-fleet 给自媒体运营配一支不糊弄读者的 AI 编辑部（每个事实可溯源、每张图有权利记录、每句话过语法门）。两者共享的机制：角色制 + 红线、确定性门禁优先于模型判断、append-only 的过程记录、两账本隔离（内部工作笔记 vs 面向公众的产出）。

## 双运行时

- **Claude Code**：整仓库即插件。`agents/` 提供八个子代理，`commands/press-init` 一键建工作区，`skills/press-ops` 承载全部协议文档。
- **Codex**：`skills/press-ops/` 目录自包含，拷入项目 `.agents/skills/press-ops/` 即用；单会话按 `references/team.md` 依次扮演各角色（这正是本项目诞生时的运行方式）。

核心设计约束：**协议文档只有一份**（`skills/press-ops/references/`），两个运行时都指向它，避免双源漂移。

## 三阶段路线图

### 第一阶段（现在）：内容与流量

开源仓库本身是账号的差异化资产——"明牌编辑部"是别人抄不走的定位，因为抄走的只是 SOP，抄不走公开运行它的过程记录。这一阶段的唯一目标是把内容和流量做起来：公众号长文全流程跑顺，X 改写管线补齐工具，每篇文末标注 SOP 版本 + 仓库链接，把读者评论当作第一批外部信号灌进复盘层。

### 第二阶段：工具补全与多平台

- v0.2：X 发布适配器（默认 dry-run，`--execute` 门禁同微信）+ 指标回拉进复盘层。
- v0.3：小红书卡片管线（确定性模板出图，导出待发包，不做非官方自动发布）。
- 复盘数据公开化：每篇文章的 retro 摘要进仓库，形成公开的"这套 SOP 到底行不行"的证据链。
- **跨模型客座审稿**（已提供协议，见 `skills/press-ops/references/guest-audit.md`）：让不同模型家族的审计员冷读同一篇稿。这是互评的单机版——在没有外部舰队时先把交换格式跑通，也规避"同门模型互审盲点重合"的问题。

### 第三阶段（有触发条件）：SOP 社区 / 舰队网络

愿景：多个创作者各自用 press-fleet 跑自己的号，用同一套缺陷分类法（`FACT-*` / `HUMAN-AI-*` / `COPY-*`…）和交接协议**互相审稿**——A 家的 release-auditor 冷读 B 家的内容包，交换审计报告。因为门禁产物本身是结构化的，跨舰队互评可以大部分自动化，这是"AI 互评社区"有实质内容的版本：不是 AI 互相点赞，是 AI 编辑部互相挑刺。

**明确的触发条件（在此之前只做协议设计，不建平台）**：

1. 本账号有稳定读者盘（社区是流量的果，不是因）；
2. 至少 2-3 个外部团队真实在用这套 SOP 跑自己的号；
3. 跨舰队审计协议在两家之间人肉跑通过至少一轮。

起步形态是 GitHub org + 一份交换格式约定，不是网站。

## 公开与私有的边界

| 公开（仓库内） | 私有（各自工作区） |
|---|---|
| SOP、角色、门禁、模板、工具 | 未发布的稿件与调研笔记 |
| 雷达方法 + 示例实例 | 平台凭据（`.env`） |
| CHANGELOG、复盘摘要 | 个人商业策略、读者个体数据 |

## 诚实边界（v0.1）

- 微信管线：完整、经真实运行验证。
- X 管线：改写方法成熟，发布工具未打包。
- 小红书管线：仅设计约定。
- 跨舰队互评：仅愿景与触发条件。

框架自己也遵守自己的规矩：没跑通的东西标 roadmap，不标 feature。
